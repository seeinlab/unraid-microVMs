## 1. Path Organization (No Conflicts)

### Binaries (ephemeral, copied from flash each boot)
```
/usr/local/bin/
├── cloud-hypervisor          # Already installed (v52.0)
├── ch-remote                 # Already installed (v52.0)
├── firecracker               # Already installed (v1.16.0)
├── crane                     # Already installed (v0.21.7)
├── ttyd                      # Already installed (v1.7.7)
├── microvm-console           # Already installed
├── containerd-microvm        # NEW: separate from Docker's /usr/bin/containerd
├── ctr-microvm               # NEW: containerd CLI (namespaced)
└── flintlockd                # NEW: gRPC microVM orchestrator
```

**Why `containerd-microvm` not `containerd`?**
- Docker owns `/usr/bin/containerd` → never overwrite
- Our instance is completely separate (different socket, state, config)
- Named distinctly to avoid confusion

### Persistent Cache (flash, survives reboot)
```
/boot/config/plugins/microvm.manager/
├── cloud-hypervisor          # Existing
├── ch-remote                 # Existing
├── firecracker               # Existing
├── crane                     # Existing
├── ttyd                      # Existing

├── ctr-microvm               # NEW (~25MB)
├── flintlockd                # NEW (~30MB)
└── microvm.manager.cfg       # Existing config
```

## 2. Conflict Analysis

| Resource | Docker | microVM Manager | Conflict? |
|----------|--------|-----------------|-----------|
| containerd binary | `/usr/bin/containerd` | `/usr/local/bin/containerd-microvm` | ❌ No |
| containerd socket | `/var/run/docker/containerd/containerd.sock` | `/run/microvm/containerd.sock` | ❌ No |
| containerd state | `/var/lib/docker/containerd/` | `/mnt/user/microvms/containerd/` | ❌ No |
| Storage driver | overlay2 | devmapper thin-pool | ❌ No |
| /dev/mapper | `control` only | `microvm-thinpool` | ❌ No |
| Loopback devices | loop0-3 used | loop10+ (use `--find`) | ❌ No |
| Network (TAP) | docker0 bridge | br0 bridge | ❌ No |
| Port 9090 | unused | flintlockd gRPC | ❌ No |
| KVM | libvirt VMs | CH/FC microVMs | ⚠️ Shared (OK) |

**Verdict: ZERO conflicts.** All paths are distinct.

## 3. Containerd Configuration

```toml
# /run/microvm/containerd-config.toml
# Generated by rc.microvm at boot time

version = 2

# Persistent state on array (OCI layers, metadata)
root = "/mnt/user/microvms/containerd"
# Ephemeral runtime state
state = "/run/microvm/containerd"

[grpc]
  address = "/run/microvm/containerd.sock"

# Use devmapper snapshotter for thin provisioning
[plugins."io.containerd.snapshotter.v1.devmapper"]
  pool_name = "microvm-thinpool"
  root_path = "/mnt/user/microvms/containerd/snapshotter/devmapper"
  base_image_size = "10GB"
  discard_blocks = true
  fs_type = "ext4"

# Disable unnecessary plugins to reduce overhead
[plugins."io.containerd.grpc.v1.cri"]
  disable = true
```

## 5. Boot Sequence (rc.microvm additions)

```bash
# === Phase 1: Thin Pool Setup ===
setup_thinpool() {
  local POOL_DIR="/mnt/user/microvms/thinpool"
  local POOL_NAME="microvm-thinpool"
  
  # Skip if already active
  dmsetup info $POOL_NAME &>/dev/null && return 0
  
  # Load kernel module
  modprobe dm_thin_pool || { log "ERROR: dm_thin_pool module failed"; return 1; }
  
  # Create sparse backing files (only once, first boot)
  mkdir -p "$POOL_DIR"
  [ -f "$POOL_DIR/data" ] || truncate -s 50G "$POOL_DIR/data"
  [ -f "$POOL_DIR/meta" ] || truncate -s 500M "$POOL_DIR/meta"
  
  # Attach loopback (use high numbers to avoid Docker's loop0-3)
  local DATA_DEV=$(losetup --find --show "$POOL_DIR/data")
  local META_DEV=$(losetup --find --show "$POOL_DIR/meta")
  local DATA_SECTORS=$(blockdev --getsize "$DATA_DEV")
  
  # Create thin pool
  dmsetup create "$POOL_NAME" \
    --table "0 $DATA_SECTORS thin-pool $META_DEV $DATA_DEV 128 32768 1 skip_block_zeroing"
  
  log "Thin pool ready: /dev/mapper/$POOL_NAME"
}

# === Phase 2: Start Containerd ===
start_containerd() {
  mkdir -p /run/microvm /mnt/user/microvms/containerd
  
  # Generate config
  cat > /run/microvm/containerd-config.toml << 'EOF'
  ... (see section 3 above)
  EOF
  
  # Start with our config (separate from Docker's)
  containerd-microvm --config /run/microvm/containerd-config.toml &
  echo $! > /run/microvm/containerd.pid
  
  # Wait for socket
  local waited=0
  while [ ! -S /run/microvm/containerd.sock ] && [ $waited -lt 10 ]; do
    sleep 1; waited=$((waited+1))
  done
}

# === Phase 3: Start Flintlockd ===
start_flintlockd() {
  flintlockd run \
    --containerd-socket=/run/microvm/containerd.sock \
    --grpc-endpoint=0.0.0.0:9090 \
    --parent-iface=br0 \
    --default-provider=cloudhypervisor \
    --insecure &
  echo $! > /run/microvm/flintlockd.pid
}
```

## 6. Shutdown Sequence

```bash
stop_flintlockd() {
  [ -f /run/microvm/flintlockd.pid ] && kill $(cat /run/microvm/flintlockd.pid) 2>/dev/null
  rm -f /run/microvm/flintlockd.pid
}

stop_containerd() {
  [ -f /run/microvm/containerd.pid ] && kill $(cat /run/microvm/containerd.pid) 2>/dev/null
  rm -f /run/microvm/containerd.pid /run/microvm/containerd.sock
}

teardown_thinpool() {
  # Deactivate all thin volumes first
  dmsetup ls --target thin | awk '{print $1}' | while read vol; do
    dmsetup remove "$vol" 2>/dev/null
  done
  # Remove pool
  dmsetup remove microvm-thinpool 2>/dev/null
  # Detach loops
  losetup -D  # only detaches unused ones (Docker's are still in use)
}
```

## 7. Installation Script (one-time)

```bash
#!/bin/bash
# install-flintlock.sh - Download and cache binaries on flash

CACHE="/boot/config/plugins/microvm.manager"
ARCH="amd64"

echo "=== Downloading containerd ==="
CTRD_VER="1.7.24"
wget -qO /tmp/containerd.tar.gz \
  "https://github.com/containerd/containerd/releases/download/v${CTRD_VER}/containerd-${CTRD_VER}-linux-${ARCH}.tar.gz"
tar -xzf /tmp/containerd.tar.gz -C /tmp/ bin/containerd bin/ctr
mv /tmp/bin/containerd "$CACHE/containerd-microvm"
mv /tmp/bin/ctr "$CACHE/ctr-microvm"
rm -rf /tmp/containerd.tar.gz /tmp/bin
chmod +x "$CACHE/containerd-microvm" "$CACHE/ctr-microvm"

echo "=== Downloading flintlockd ==="
FL_VER="0.9.0"
wget -qO "$CACHE/flintlockd" \
  "https://github.com/liquidmetal-dev/flintlock/releases/download/v${FL_VER}/flintlockd"
chmod +x "$CACHE/flintlockd"

echo "=== Done ==="
ls -lh "$CACHE/containerd-microvm" "$CACHE/ctr-microvm" "$CACHE/flintlockd"
echo "Total: $(du -sh $CACHE | cut -f1)"
```

## 8. How Plugin Uses Flintlock

### Current (Direct mode — no flintlock)
```
WebGUI → MicroVMAdmin.php → rc.microvm → cloud-hypervisor/firecracker CLI
```

### With Flintlock (Orchestrated mode)
```
WebGUI → MicroVMAdmin.php → flintlockd gRPC API → containerd → cloud-hypervisor/firecracker
```

### API Translation
| Plugin Action | Direct Mode (current) | Flintlock Mode |
|--------------|----------------------|----------------|
| Create VM | crane export + mkfs + mount | `fl microvm create --image=docker.io/nginx:alpine` |
| Start VM | `cloud-hypervisor --kernel...` | gRPC CreateMicroVM (auto-boots) |
| Stop VM | `ch-remote power-button` | gRPC DeleteMicroVM |
| Snapshot | `ch-remote pause+snapshot` | Direct (flintlock doesn't manage snapshots) |

### Hybrid Approach (recommended)
- Keep **Direct mode** as default (simple, proven, no extra daemons)
- Add **Flintlock mode** as optional "Advanced" in Settings
- When enabled: start containerd + flintlock on service start
- Benefits: OCI image layer sharing, instant thin cloning, gRPC API

## 10. Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Containerd conflicts with Docker | Separate binary, socket, state, config |
| Thin pool fills up | Monitor in Usage Statistics tab; auto-extend |
| Loopback conflicts | Use `losetup --find` (auto-selects free) |
| Boot order (array not ready) | `wait_for_array()` already in rc.microvm |
| Binary size (100MB+) | Store on flash, copy to RAM each boot |
| Flintlock crashes | Plugin falls back to Direct mode |
| Unraid update breaks dm-thin | Module is standard kernel, unlikely |
